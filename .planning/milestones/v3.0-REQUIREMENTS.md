# Arquivo de Histórico — REQUIREMENTS Milestone 3 (v3.0)

Este documento registra os Requisitos Funcionais, Regras de Negócio e Casos de Uso que foram 100% satisfeitos e validados ao término da **Milestone 3** do PECAÊ.

---

## Requisitos Funcionais Satisfeitos

### M05 — Cadastro de Sucata / Veículo
- [x] **RF40:** Vendedor deve poder cadastrar sucata completa via wizard multi-step com: versão do veículo, ano, cor, localização, observações e placa opcional (mascarada) (P0)
- [x] **RF41:** Vendedor deve poder marcar quais peças estão disponíveis usando lista fixa de PartCategory (P0)
- [x] **RF42:** Vendedor deve poder fazer upload de 4 a 10 fotos por sucata (P0)
- [x] **RF43:** Sistema deve submeter todo anúncio novo ou editado para fila de moderação com status PENDING (RN14) (P0)
- [x] **RF44:** Vendedor deve poder editar sucata existente (dados e fotos) (P0)
- [x] **RF45:** Vendedor deve poder atualizar lista de peças disponíveis de forma rápida (P0)
- [x] **RF46:** Vendedor deve poder marcar sucata como 'Vendida' encerrando o anúncio (P0)

---

## Regras de Negócio Implementadas

### M05 — Cadastro de Sucata / Veículo
- [x] **RN04:** Proibido solicitar, registrar ou expor valor ou preço para anúncios de sucatas (lotes completos).
- [x] **RN05:** Proibido expor ou solicitar número de chassi ou código VIN para lotes de sucata.
- [x] **RN06:** Marcação de sucata como vendida (SOLD) é definitiva; impede o retorno para DRAFT ou ACTIVE e encerra síncrona o anúncio correspondente.
- [x] **RN10:** Verificação de duplicidade de placa e anúncios similares do mesmo vendedor no mesmo modelo/ano.
- [x] **RN14:** Toda criação ou edição de lote de sucata força o status do anúncio e do veículo de volta para PENDING, exigindo nova rodada de aprovação de segurança da moderação.
- [x] **RN-M05-A:** *Opção A de Concorrência:* Expurgar e cancelar de forma ativa e atômica da fila de jobs do BullMQ/Redis qualquer job anterior para o mesmo `vehicleId` antes de agendar o novo lote de processamento de fotos.
- [x] **RN-M05-B:** *Opção B de Falhas:* Se o processamento assíncrono falhar no Sharp/BullMQ, reverter de forma atômica o status do veículo e anúncio para `DRAFT` na base de dados e deletar fisicamente quaisquer arquivos intermediários criados no Supabase Storage.

- [x] **UC21:** Editar sucata existente (via wizard no modo edit e salvamento via PATCH).
- [x] **UC22:** Marcar sucata como vendida (ação rápida que inativa o anúncio e define como SOLD).
